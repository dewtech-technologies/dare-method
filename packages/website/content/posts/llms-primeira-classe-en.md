# LLMs as First-Class Citizens in DARE v3.0

When most teams talk about "AI-assisted development," they mean something simple: open Claude or GPT, paste a problem, copy the answer. The LLM is an external tool — powerful, but completely outside the mental model of how software is actually built.

DARE v3.0 changes that. Instead of treating LLMs as development support tools, the method elevates them to **first-class citizens** — both in the construction process and in the architecture of the resulting product.

## The Problem with the Current Approach

Imagine you're building a product that uses AI to classify legal documents. The typical flow today:

1. You open Claude
2. Type "classify this document as X, Y, or Z"
3. Receive a response
4. Paste it somewhere in the code

What happens when you need to:
- Understand why a specific document was misclassified?
- Swap Claude for GPT-4 without rewriting half the system?
- Limit costs when volume scales from 100 to 100,000 documents/day?
- Audit every classification decision for a legal proceeding?

The honest answer is: you can't, or you can with a lot of pain, because the LLM was treated as a black box glued ad-hoc into the system.

## What "First-Class Citizen" Means in DARE

In DARE v3.0, an LLM integrated into the product follows the same rigor as any other architectural component — a database, a message queue, an external API. This manifests in three dimensions:

### 1. Typed Contracts (Prompt Templates)

Prompts are not magic strings. They are contracts with typed inputs, validated outputs, and semantic versioning.

```ruby
class ClassifyLegalDocumentPrompt < Dare::Prompt::Base
  param :document_text, type: String
  param :document_type,  type: String, enum: %w[contract petition ruling]
  param :jurisdiction,   type: String, default: "US"

  output_schema do
    field :category,    type: String, enum: %w[civil criminal administrative]
    field :confidence,  type: Float,  min: 0.0, max: 1.0
    field :reasoning,   type: String
  end

  template <<~PROMPT
    You are a legal document classifier.

    Document type: {{document_type}}
    Jurisdiction: {{jurisdiction}}

    Classify the following document as civil, criminal, or administrative.
    Respond in JSON following the schema exactly.

    Document:
    {{document_text}}
  PROMPT
end
```

The output schema is validated automatically. If the response doesn't respect the contract, the `dare-llm-integration` gate will fail the task before it's marked DONE.

### 2. Token Budgeting

Tokens cost money. DARE treats the token budget as a first-class architectural constraint, configured in `.dare/config.json`:

```json
{
  "skills": {
    "dare-llm-integration": {
      "token_budget": {
        "per_request_max": 4096,
        "per_day_max": 500000,
        "alert_at_percent": 80,
        "hard_limit": true
      }
    }
  }
}
```

With `hard_limit: true`, the system refuses calls that would exceed the daily budget — instead of letting the bill explode at the end of the month.

### 3. End-to-End Observability

Every LLM call in DARE is automatically traced with OpenTelemetry:

```ruby
# Automatic trace via dare-llm-integration + dare-quality-telemetry
result = Dare::LLM.call(
  prompt: ClassifyLegalDocumentPrompt.new(
    document_text: doc.text,
    document_type: doc.type
  ),
  model: "claude-sonnet-4-6"
)

# In your trace dashboard:

#   prompt_tokens: 842
#   completion_tokens: 156

#   cache_hit: false
#   document_type: contract
```

## How DARE v3.0 Uses LLMs in the Development Process Itself

The most powerful feature of DARE v3.0 is applying these same patterns to the development process itself.

In the DESIGN and ARCHITECT phases, the CLI injects contextual prompts that include:
- The project's `llms.txt` (generated by `dare-ax`)
- Active skills and their patterns
- Architectural decision history (previous BLUEPRINT.md)

This means the AI generating your next BLUEPRINT doesn't start from scratch — it knows the accumulated context of your project, the patterns you've already adopted, and the constraints you've established.

```bash
dare blueprint DARE/DESIGN.md
# Context loaded:
#   llms.txt — 3.2k tokens
#   skills: dare-layered-design, dare-llm-integration, dare-ax
#   Previous BLUEPRINT: 8 indexed decisions
#   Generating BLUEPRINT.md...
```

## The Ralph Loop and Prompt Self-Correction

A lesser-discussed aspect of the Ralph Loop is that it also corrects prompts.

If a regression test fails because an LLM's output format changed, the Ralph Loop:

1. Detects the failure in the schema validation gate
2. Analyzes the diff between expected and received output
3. Proposes an adjustment to the prompt template
4. Re-executes until the gate passes

This transforms prompts from fragile artifacts into resilient components that evolve alongside the system.

## Conclusion

Treating LLMs as second-class citizens — support tools bolted outside the architectural model — creates silent technical debt that will exact its price when the system scales.

DARE v3.0 solves this by bringing the same engineering principles we apply to databases, queues, and APIs to LLM integration: typed contracts, explicit budgets, total observability.

The result is a system where you know exactly how much each feature cost in tokens, can audit every AI decision, and can swap the underlying model without rewriting code.
