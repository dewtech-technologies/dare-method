# RFC: AGPL v3 Adoption in DARE CLI

> **Status:** Open for Discussion  
> **Date Created:** 2026-05-14  
> **Target Decision:** 2026-05-28  
> **Target Release:** v3.0.0 (early June 2026)

---

## TL;DR

**Proposal:** Migrate `@dewtech/dare-cli` from **MIT** to **AGPL v3** in v3.0.0.

**When:** June 2026 (after 3 features in MIT: Excalidraw, Ruby on Rails, Flutter)

**Impact:**
- ✅ Your generated project: Remains MIT (yours)
- ✅ Using internally: Zero impact
- ⚠️ Selling "DARE Cloud SaaS": Must be AGPL or commercial license

**Business Model:** Open Core
- CLI (AGPL) → Free, community-controlled
- Desktop/Agents (Proprietary) → Paid, monetized

---

## 1. Context

### What is DARE?

[DARE Framework](https://github.com/dewtech-technologies/dare-method) is an AI-driven development methodology (Design → Architect → Review → Execute) featuring:

- **Methodology** (MIT) — documented, open, reusable
- **CLI** (currently MIT) — executable tool
- **Implementations** (MIT) — skills for Cursor, Claude Code, Antigravity
- **Future products** (Proprietary) — Desktop, Orchestration, Services

### Why MIT today?

Launched as MIT because:
1. Methodology is the differentiator, not the tool
2. Focus on rapid adoption and community growth
3. Commercial products weren't planned yet

### Why AGPL v3 now?

**Facts:**
- CLI grew significantly (v0 → v2.15.0 in ~6 months)
- Roadmap includes complementary products (Desktop, Agents)
- Risk: competitors copy CLI as "DARE SaaS" without contributing

**Goal:** Protect investment + ensure contributions return to community.

## 2. Proposal

### Move to AGPL v3

**License file:**
```
LICENSE → AGPL v3 content
```

**Package.json:**
```json
{
  "license": "MIT" → "license": "AGPL-3.0-only",
  "name": "@dewtech/dare-cli"
}
```

**Version:** v3.0.0 (semantic breaking change)

### What DOES NOT change

- ✅ DARE Methodology → Remains MIT
- ✅ Documentation → Remains MIT
- ✅ Code GENERATED by CLI → MIT (yours)
- ✅ Templates → MIT (yours)
- ✅ Examples → MIT
- ❌ Executable CLI → AGPL v3

## 6. FAQ

### Q: Does my generated code become AGPL?

**A:** No. `dare init` generates **MIT**:
```
dare init my-project
├── apps/api/     ← MIT (yours)
├── apps/web/     ← MIT (yours)
└── DARE/         ← AGPL (reference)
```

You own your code. You can sell it, close it, do whatever you want.

---

### Q: Do I need to contribute to DARE?

**A:** Depends on how you use it:

| Usage | Need to Contribute? |
|-------|-----|
| Use CLI locally | ❌ No |
| Use CLI internally (your EC2/K8s) | ❌ No |
| Modify CLI and keep private | ❌ No |
| **Sell "DARE as a service"** | ✅ **Yes** (AGPL or license) |

---

### Q: When will the commercial license be available?

**A:** After v3.0.0. Terms and pricing will be defined in June based on feedback.

---

### Q: Can I use v2.15.0 (MIT) forever?

**A:** Yes. v2.15.0 and earlier remain MIT. But:
- No security updates
- No new features
- No support

We recommend upgrading to v3.0.0+ AGPL.

---

### Q: Will you abandon the project?

**A:** No. AGPL means exactly the opposite:
- ✅ Public code
- ✅ Community controls it
- ✅ Improvements come back
- ✅ Continuous development

If we abandon it, anyone can fork it.

---

### Q: What's the real reason?

**A:** Honest:
1. CLI grew beyond expectations
2. We want to monetize complementary products (Desktop)
3. We want to ensure improvements come back (no unpaid forks)
4. We want legal protection against SaaS competitors

**This is legitimate.** Open source + sustainable business model = better than MIT that dies.

## 7. Alternatives Considered

| Alternative | Why Rejected |
|---|---|
| **Stay MIT** | No protection against "DARE Cloud" competitors; hard to monetize Desktop |
| **Apache 2.0** | Patent protection but doesn't solve SaaS loophole |
| **GPL v3** | Doesn't cover network use; SaaS can still circumvent |
| **Dual MIT + Commercial** | Too complex now; scale with AGPL later |
| **Proprietary** | Against DARE philosophy; community wouldn't trust |
| **FSL (Fair Source)** | Good but less recognized; AGPL is better known |
